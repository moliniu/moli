# Admin Console Vue3 Element Plus TS

这是一个“能把模型输出钉死”的后台 Skill，不是科普文。

你必须把回答做成可落地的工程决策：文件落点、组件边界、类型边界、权限口径、状态语义、样式收口。任何只讲理念不落地的输出都算失败。

## 适用范围

- Vue3 + Element Plus + TypeScript 的后台管理端。
- 列表页 / 详情页 / 表单页 / 仪表盘。
- 路由权限、按钮权限、请求封装、Pinia、表格表单、布局。

## 不适用范围

- uni-app、小程序、App。
- 营销官网、活动页、品牌站。
- 仅做 UI 点评但不涉及工程结构。

## 触发规则（硬规则）

- 若用户明确提到：后台/管理端/控制台/运营端/Element Plus/Pinia/权限路由/表格表单，则进入本 Skill。
- 若用户明确提到：uni-app/小程序/App/pages.json/分包/条件编译/wot，则拒绝并引导到 `uniapp-vue3-wot-design-uni`。
- 若用户同时提到后台与小程序：先要求拆分为两个独立需求，再分别回答。

## 进入前必问（硬规则）

缺一项就必须先追问，最多追问 2 轮，仍缺则只输出“最小落地骨架 + 风险点”，禁止写大段方案。

必问三项：

- 页面类型：列表 / 详情 / 表单 / 仪表盘
- 数据来源：已有 API / 需要定义 API / 临时 Mock
- 权限：页面级是否受控、按钮级权限点命名规则是否已存在

若是存量项目，再补问两项：

- 增量新页面还是旧页面重构
- 是否已有统一 request 层与错误语义

## 输出顺序（硬规则）

正式回答必须按以下顺序输出，顺序不允许打乱：

1. 文件落点
2. 组件拆分
3. 类型与数据模型
4. 状态语义（按页面类型）
5. 权限与路由
6. 样式收口（tokens）
7. 最小代码骨架（仅在必要时）
8. 风险与验收点

## 输出契约（必须满足）

每次回答至少包含：

- 文件落点：精确到目录与文件名
- 组件拆分：页面/业务组件/base 组件边界
- 类型策略：DTO/VO/Query/Form/Enum 的放置点
- 状态语义：loading/empty/error 的可恢复动作
- 权限闭环：路由级 + 页面级 + 按钮级
- 样式收口：tokens 文件与禁止散写规则

若用户要求“只要一段代码”：

- 允许给最小片段，但仍必须给文件落点 + 权限/状态风险点。

## 页面类型的状态语义（硬规则）

- 列表页：必须覆盖 loading/empty/error/pagination；empty 必须区分无数据与查询无结果；error 必须有重试。
- 详情页：必须覆盖 loading/error；empty 必须区分不存在与无权限。
- 表单页：必须覆盖提交 loading、校验错误、接口错误、离开页面未保存提示（若涉及编辑）。
- 仪表盘：必须覆盖局部 loading、分块错误隔离（某块挂了不影响整页）。

## 权限闭环（硬规则）

若用户未提供权限信息：

- 必须追问“是否有按钮级权限”。
- 若用户拒答：默认页面受控、按钮收敛到最小集合，并在风险里说明。

按钮级权限点命名要求：

- 必须是稳定字符串，不允许临时拼接。
- 必须能映射到后端/配置中心的权限点。

路由级权限要求：

- 必须说明路由 meta 承载点（如 meta.perm）。
- 必须说明无权限时的降级去向（403 或重定向）。

## 目录与命名

标准目录结构：

```text
src/
  api/
  assets/
  components/
    base/
    biz/
  composables/
  layouts/
  router/
    routes/
    guard.ts
  stores/
  styles/
    tokens.css
    index.scss
  types/
  utils/
    request.ts
  views/
  main.ts
  App.vue
```

命名硬规则：

- 目录：kebab-case
- 组件：PascalCase
- Base 组件：BaseXxx
- composable：useXxx
- store：useXxxStore
- API：xxx.api.ts

## 样式收口（硬规则）

- 颜色/字号/间距必须走 tokens 或变量层。
- 若用户给了设计稿：只允许在 tokens 中新增变量，不允许页面散写。

## 冲突仲裁规则

当“快”与“规范”冲突时：

- 优先保证：文件落点 + 状态语义 + 权限闭环。
- 允许牺牲：长代码、长教程、全量目录说明。

当“跨端混提”与“单端稳定输出”冲突时：

- 必须拆分需求。
- 不允许混答。

## 反例（禁止输出的样子）

- 只讲理念：例如“建议使用分层架构、注意状态管理”但不给任何文件落点与边界。
- 权限一句带过：例如“加一下权限控制”但不说路由 meta、按钮点命名与降级路径。
- 列表页只给表格：没有分页、没有 empty/error、没有重试动作。

## Vue2 存量策略与禁忌

- 只允许增量策略：双工程并存、纯 TS 内核复用。
- 禁止许诺整站无痛迁移。
- 禁止把 Vue2 历史结构写成“未来规范”。

## 质量检查清单（验收版）

- 是否在前两段就给出文件落点
- 是否明确页面类型并按类型给状态语义
- 是否说明按钮级权限点策略与路由 meta 承载点
- 是否给出 error 的恢复动作（重试/去登录/提示）
- 是否把样式收口到 tokens 并明确禁止散写
- 是否避免混答 uni-app 需求
