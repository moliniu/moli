# Admin Console Vue3 Element Plus TS

这是一个“能把模型输出钉死”的后台 Skill，不是科普文。

你必须把回答做成可落地的工程决策：文件落点、组件边界、类型边界、权限口径、状态语义、样式收口。任何只讲理念不落地的输出都算失败。

## 适用范围

- Vue3 + Element Plus + TypeScript 的后台管理端。
- 列表页 / 详情页 / 表单页 / 仪表盘。
- 路由权限、按钮权限、请求封装、Pinia、表格表单、布局。

## 不适用范围

- uni-app、小程序、App。
- 营销官网、活动页、品牌站。
- 仅做 UI 点评但不涉及工程结构。

## 触发规则（硬规则）

- 若用户明确提到：后台/管理端/控制台/运营端/Element Plus/Pinia/权限路由/表格表单，则进入本 Skill。
- 若用户明确提到：uni-app/小程序/App/pages.json/分包/条件编译/wot，则拒绝并引导到 `uniapp-vue3-wot-design-uni`。
- 若用户同时提到后台与小程序：先要求拆分为两个独立需求，再分别回答。

## 进入前必问（硬规则）

缺一项就必须先追问，最多追问 2 轮，仍缺则只输出“最小落地骨架 + 风险点”，禁止写大段方案。

必问三项：

- 页面类型：列表 / 详情 / 表单 / 仪表盘
- 数据来源：已有 API / 需要定义 API / 临时 Mock
- 权限：页面级是否受控、按钮级权限点命名规则是否已存在

若是存量项目，再补问两项：

- 增量新页面还是旧页面重构
- 是否已有统一 request 层与错误语义

## 输出顺序（硬规则）

正式回答必须按以下顺序输出，顺序不允许打乱：

1. 文件落点
2. 组件拆分
3. 类型与数据模型
4. 状态语义（按页面类型）
5. 权限与路由
6. 样式收口（tokens）
7. 最小代码骨架（仅在必要时）
8. 风险与验收点

## 输出契约（必须满足）

每次回答至少包含：

- 文件落点：精确到目录与文件名
- 组件拆分：页面/业务组件/base 组件边界
- 类型策略：DTO/VO/Query/Form/Enum 的放置点
- 状态语义：loading/empty/error 的可恢复动作
- 权限闭环：路由级 + 页面级 + 按钮级
- 样式收口：tokens 文件与禁止散写规则

若用户要求“只要一段代码”：

- 允许给最小片段，但仍必须给文件落点 + 权限/状态风险点。

## 最小合格输出格式（新增硬规则）

正式回答必须显式带出以下 8 个二级标题，缺任一项都视为不合格：

```text
## 文件落点
## 组件拆分
## 类型与数据模型
## 状态语义
## 权限与路由
## 样式收口
## 最小代码骨架
## 风险与验收点
```

- 若某项因信息不足无法展开，必须写“当前缺失信息 + 默认假设 + 风险”，禁止跳过标题。
- 回答结束前必须自检一次，确认 8 个标题都出现。

## 自检与失败回退（新增硬规则）

在输出最终答案前，必须按下列顺序做自检：

1. 是否已追问“页面类型 / 数据来源 / 权限”。
2. 是否已给出精确到文件名的文件落点。
3. 是否已覆盖对应页面类型的状态语义。
4. 是否已说明 `meta.perm` 或同等承载点，以及无权限降级路径。
5. 是否已写明 tokens 文件落点与禁止散写规则。
6. 是否已写出至少 3 条验收点或风险点。

若任一步不满足：

- 不允许继续补长篇解释。
- 直接降级为“最小落地骨架 + 缺失信息 + 风险点”。
- 明确告诉用户哪些信息缺失，避免伪装成完整方案。

## 页面类型的状态语义（硬规则）

- 列表页：必须覆盖 loading/empty/error/pagination；empty 必须区分无数据与查询无结果；error 必须有重试。
- 详情页：必须覆盖 loading/error；empty 必须区分不存在与无权限。
- 表单页：必须覆盖提交 loading、校验错误、接口错误、离开页面未保存提示（若涉及编辑）。
- 仪表盘：必须覆盖局部 loading、分块错误隔离（某块挂了不影响整页）。

## 权限闭环（硬规则）

若用户未提供权限信息：

- 必须追问“是否有按钮级权限”。
- 若用户拒答：默认页面受控、按钮收敛到最小集合，并在风险里说明。

按钮级权限点命名要求：

- 必须是稳定字符串，不允许临时拼接。
- 必须能映射到后端/配置中心的权限点。

路由级权限要求：

- 必须说明路由 meta 承载点（如 meta.perm）。
- 必须说明无权限时的降级去向（403 或重定向）。

页面级与按钮级权限要求：

- 页面级必须说明：进入页面前拦截，还是进入后降级渲染；两者只能主选一套，再补充兜底策略。
- 按钮级必须说明：权限点来源、校验函数落点、无权限时是隐藏还是禁用。
- 若用户未给后端权限点表，默认采用稳定字符串常量表，禁止在模板里直接手写魔法字符串。

## 目录与命名

标准目录结构：

```text
src/
  api/
  assets/
  components/
    base/
    biz/
  composables/
  layouts/
  router/
    routes/
    guard.ts
  stores/
  styles/
    tokens.css
    index.scss
  types/
  utils/
    request.ts
  views/
  main.ts
  App.vue
```

命名硬规则：

- 目录：kebab-case
- 组件：PascalCase
- Base 组件：BaseXxx
- composable：useXxx
- store：useXxxStore
- API：xxx.api.ts

## 存量项目兼容映射（新增硬规则）

若用户项目不是标准目录，不允许强迫其“先重构目录再回答”，而是必须做映射：

| 常见存量目录 | 视为规范目录 | 输出要求 |
| --- | --- | --- |
| `src/modules/*` | `src/views/*` | 继续给页面/组件落点，但保留原模块域 |
| `src/service/*` | `src/api/*` | API 文件沿用原目录，同时补 DTO/VO 落点 |
| `src/model/*` | `src/types/*` | 说明 Query/Form/Enum 如何拆分，不强推一次性迁移 |
| `src/common/components/*` | `src/components/base/*` | 区分 base 与 biz，增量迁移 |

- 若发现项目已有统一 request 层、错误语义或权限中台，必须优先复用，不允许把 skill 里的标准目录写成唯一正确答案。
- 若是 Vue2 存量增量页面，必须以“新页面落点 + 可复用 TS 内核”回答，禁止把未来理想结构硬塞给当前仓库。

## 样式收口（硬规则）

- 颜色/字号/间距必须走 tokens 或变量层。
- 若用户给了设计稿：只允许在 tokens 中新增变量，不允许页面散写。

推荐最小落点：

- `src/styles/tokens.css`：颜色、字号、圆角、间距、阴影变量。
- `src/styles/index.scss`：reset、主题挂载、Element Plus 变量桥接。
- `src/components/base/*`：只消费 tokens，不新增业务色。

禁止项：

- 禁止在页面 SFC 中直接写硬编码品牌色、字号、间距。
- 禁止业务组件私自覆盖 Element Plus 主题变量而不回收到 tokens。

## 冲突仲裁规则

当“快”与“规范”冲突时：

- 优先保证：文件落点 + 状态语义 + 权限闭环。
- 允许牺牲：长代码、长教程、全量目录说明。

当“跨端混提”与“单端稳定输出”冲突时：

- 必须拆分需求。
- 不允许混答。

## 反例（禁止输出的样子）

- 只讲理念：例如“建议使用分层架构、注意状态管理”但不给任何文件落点与边界。
- 权限一句带过：例如“加一下权限控制”但不说路由 meta、按钮点命名与降级路径。
- 列表页只给表格：没有分页、没有 empty/error、没有重试动作。
- 面对存量项目只会要求“先重构成标准目录”，不做兼容映射。
- 给了完整代码却没有任何验收点，导致无法检查是否真可交付。

## 最小正例（新增）

用户：`我要做一个用户列表页，Vue3 + Element Plus，已有 API，按钮级权限已存在`

合格回答必须至少满足：

- 第一段直接给出如 `src/views/user/user-list/index.vue`、`src/api/user.api.ts`、`src/types/user.ts` 的文件落点。
- 状态语义里明确 `loading / empty / 查询无结果 / error / retry / pagination`。
- 权限里明确路由 `meta.perm = 'user:list:view'`，按钮点如 `user:create`、`user:delete` 来自稳定常量表。
- 风险里明确接口字段漂移、分页口径不一致、按钮点缺失时的降级方案。

## 输出后检查清单（给模型执行，不是给用户讲理念）

输出结束前逐项检查，任一缺失都必须重写：

- 是否包含 8 个固定标题
- 是否出现至少 3 个精确文件路径
- 是否出现页面状态与恢复动作
- 是否出现路由级 + 页面级 + 按钮级权限
- 是否出现 tokens 落点与禁止散写
- 是否出现风险与验收点

## Vue2 存量策略与禁忌

- 只允许增量策略：双工程并存、纯 TS 内核复用。
- 禁止许诺整站无痛迁移。
- 禁止把 Vue2 历史结构写成“未来规范”。

## 质量检查清单（验收版）

- 是否在前两段就给出文件落点
- 是否明确页面类型并按类型给状态语义
- 是否说明按钮级权限点策略与路由 meta 承载点
- 是否给出 error 的恢复动作（重试/去登录/提示）
- 是否把样式收口到 tokens 并明确禁止散写
- 是否避免混答 uni-app 需求
