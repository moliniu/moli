# Skills 仓库

本仓库沉淀两个面向未来项目的前端 Skill，目标不是堆更多“规范文字”，而是把模型回答收敛成可复查、可回退、可扩展的工程输出协议。

它们约束四件事：

- 什么时候进入 Skill，什么时候必须拒绝或拆分需求。
- 信息不足时先追问，而不是直接进入模板化输出。
- 输出必须落到文件、组件、类型、状态、权限或平台差异、样式收口这些硬点。
- 输出不合格时必须降级回退，而不是继续用长篇解释掩盖缺项。

## 目录

- `admin-console-vue3-elementplus-ts/`：Vue3 + Element Plus + TypeScript 后台管理端。
- `uniapp-vue3-wot-design-uni/`：UniApp + Vue3 + wot-design-uni 移动端 / 小程序 / App。

## 用途

两个 Skill 都做四件事：

- 判断当前需求是否在本 Skill 适用范围内。
- 信息不足时主动追问，而不是直接进入模板化输出。
- 按固定顺序输出文件落点、组件/类型边界、状态语义、权限/平台差异、样式收口。
- 出现越界、跨端混提、散写条件编译、权限不闭环等失败场景时直接纠偏。

当前版本额外补了三层能力：

- 最小合格输出格式：固定标题和固定段落，降低漏项概率。
- 自检与失败回退：输出前自查，缺项时直接退回“最小骨架 + 风险点”。
- 存量项目兼容映射：不强迫用户先重构目录，再回答当前问题。

## 适用范围对照

| Skill | 适用 | 不适用 |
| --- | --- | --- |
| `admin-console-vue3-elementplus-ts` | Vue3 后台管理系统、列表/详情/表单/仪表盘、权限路由、表格表单、请求封装、Pinia | uni-app / 小程序 / App、营销官网、纯 UI 点评 |
| `uniapp-vue3-wot-design-uni` | 微信小程序、uni-app App、H5、`pages.json`、分包、平台适配、wot-design-uni | 后台 Web 管理端、普通 Vue3 Web、纯 UI 点评 |

## 使用方法

### 1. 触发

- 用户提到“后台 / 管理端 / 控制台 / Element Plus / Pinia / 权限路由 / 表格表单”，使用 `admin-console-vue3-elementplus-ts`。
- 用户提到“uni-app / 小程序 / App / pages.json / 分包 / 条件编译 / wot-design-uni / APP-PLUS”，使用 `uniapp-vue3-wot-design-uni`。
- 同时提到两端：先要求拆为两个独立需求，再分别调用。

### 2. 进入前必须追问

两个 Skill 都强制先问三项再输出方案：

- 后台：页面类型、数据来源、权限要求。
- 移动端：页面路径或业务域、端能力、是否分包。

### 3. 输出顺序（硬规则）

后台：

1. 文件落点
2. 组件拆分
3. 类型与数据模型
4. 状态语义（按页面类型）
5. 权限与路由
6. 样式收口（tokens）
7. 最小代码骨架（仅在必要时）
8. 风险与验收点

移动端：

1. `pages.json` 改动点
2. 平台差异收口位置
3. 请求封装与登录态
4. 页面状态语义（按页面类型）
5. wot 引入与必要配置
6. 样式 tokens 与密度
7. 最小代码骨架（仅在必要时）
8. 真机验证点与风险

### 3.1 最小合格输出格式

两个 Skill 现在不只要求“顺序正确”，还要求“段落必须显式出现”。

后台必须显式出现以下标题：

```text
## 文件落点
## 组件拆分
## 类型与数据模型
## 状态语义
## 权限与路由
## 样式收口
## 最小代码骨架
## 风险与验收点
```

移动端必须显式出现以下标题：

```text
## pages.json 改动点
## 平台差异收口位置
## 请求封装与登录态
## 页面状态语义
## wot 引入与必要配置
## 样式 tokens 与密度
## 最小代码骨架
## 真机验证点与风险
```

- 缺任何一个标题，都视为不合格输出。
- 某项信息不足时，不能省略标题，必须写“缺失信息 + 默认假设 + 风险”。

### 3.2 输出前自检与失败回退

两个 Skill 都新增了“先自检，再交付”的硬规则：

- 若没有追问关键入参，不允许装作方案完整。
- 若没有文件落点、状态语义、权限闭环或平台差异落点，不允许继续补长篇解释。
- 若缺少风险点或验收点，视为不可交付。

失败时统一回退为：

- 最小落地骨架
- 当前缺失信息
- 默认假设
- 风险点

### 4. 跨端混提

- 一律拆分需求，禁止一个回答里同时答后台和移动端。
- 后台部分走 `admin-console-vue3-elementplus-ts`。
- 移动端部分走 `uniapp-vue3-wot-design-uni`。

### 5. 失败场景识别

下列输出视为失败，需要模型自己重答：

- 只讲理念，不给文件落点、路由配置或模块边界。
- 权限只用一句话带过，不说路由 meta、按钮权限点、降级路径。
- 平台差异只用“自行兼容”糊弄，不展开 401 / 跨端 cookie / `#ifdef` 收口。
- 业务页面里散写条件编译。
- 跨端混提时硬把两端揉成一份回答。
- 缺少固定标题，或标题顺序错乱。
- 遇到信息不足时跳过缺失项，假装方案完整。
- 面对存量项目只会要求“先重构目录”，不做兼容映射。
- 给了完整代码，却没有风险点或验收点。

## 存量项目兼容策略

两个 Skill 都不再把“标准目录”当成唯一正确答案，而是要求先做目录映射，再输出方案。

典型原则：

- 已有 `request` 层、错误语义、权限中台时，优先复用。
- 已有 `pages.config.ts`、多端构建脚本、分包生成脚本时，优先复用。
- 已有历史模块目录时，先映射到规范语义，不强推一次性迁移。
- Vue2 存量项目仍然只允许增量策略，不承诺整站无痛升级。

## 机器强校验方向

当前仓库的两个 Skill 已经具备“适合被机器校验”的基础，但 README 明确一下边界：

- 现在已实现的是“强约束文本协议”。
- 下一步应升级为“结构化输出 + 校验脚本”。
- 理想链路不是“模型尽量遵守”，而是“模型输出 -> 校验器检查 -> 不合格则打回重答”。

最适合先做机器校验的项：

- 固定标题是否齐全
- 文件路径数量是否达标
- 是否出现 `meta.perm`、`pages.json`、`utils/platform.ts` 等关键落点
- 是否覆盖 `loading / empty / error` 或 `401 / 网络错误 / 业务错误 / 登录失效`
- 是否包含风险点与验收点

推荐实现方式：

- 让模型先输出结构化 JSON
- 用 `JSON Schema` 或脚本做字段校验
- 校验通过后再渲染成 Markdown 给用户

## 当前版本重点增强

- 后台 Skill：补了最小合格输出格式、自检回退、页面级与按钮级权限要求、存量目录映射、最小正例、输出后检查清单。
- uni-app Skill：补了最小合格输出格式、自检回退、平台差异例外策略、请求与登录态最小可交付要求、存量目录映射、最小正例、输出后检查清单。
- 仓库层面：README 从“说明怎么用”升级为“说明怎么判断输出是否合格”。

## Vue2 存量策略

两个 Skill 都只允许对 Vue2 存量项目输出“增量兼容策略”：

- 双工程并存
- 纯 TS 业务内核复用
- 不承诺整站无痛迁移
- 不把 Vue2 历史结构抬升成未来规范
