# UniApp Vue3 Wot Design Uni

这是一个“能把模型输出钉死”的 UniApp Skill，不是科普文。

你必须把回答做成跨端工程决策：`pages.json` 改动、平台差异收口、条件编译量化、请求语义、登录态、移动端特有状态、wot 引入方式。任何只讲组件不讲落点的输出都算失败。

## 适用范围

- uni-app Vue3 项目：微信小程序、App、H5。
- `pages.json`、分包、平台适配、请求封装、wot-design-uni 使用。
- 移动端列表/详情/表单/设置页。

## 不适用范围

- 后台 Web 管理端
- 普通 Vue3 Web 项目
- 仅做 UI 点评但不涉及 uni-app 结构

## 触发规则（硬规则）

- 若用户明确提到：uni-app/小程序/App/pages.json/分包/条件编译/wot-design-uni/APP-PLUS，则进入本 Skill。
- 若用户明确提到：后台/管理端/Element Plus/Pinia/权限路由，则拒绝并引导到 `admin-console-vue3-elementplus-ts`。
- 若同时提到：先要求拆为两个独立需求，再分别回答。

## 进入前必问（硬规则）

缺一项就必须先追问，最多追问 2 轮，仍缺则只输出“最小落地骨架 + 风险点”，禁止写大段方案。

必问三项：

- 页面路径或业务域
- 端能力：登录、分享、定位、蓝牙、相机、推送等
- 是否分包，或已有分包规则

若是存量项目，再补问两项：

- 新增页面还是旧页面改造
- 是否已有统一 request 层与平台适配层

## wot 引入方式（必须二选一）

不允许让用户“自行选择”，必须给出推荐路径：

- 推荐：uni_modules 方式。覆盖式升级，easycom 自动接入。
- 仅在以下情况选 npm：项目已统一使用 npm + Vite 显式依赖管理；或团队明确要求统一依赖锁版。

若两者都未明确：默认走 uni_modules。

## 输出顺序（硬规则）

正式回答必须按以下顺序输出，顺序不允许打乱：

1. `pages.json` 改动点
2. 平台差异收口位置
3. 请求封装与登录态
4. 页面状态语义（按页面类型）
5. wot 引入与必要配置
6. 样式 tokens 与密度
7. 最小代码骨架（仅在必要时）
8. 真机验证点与风险

## 最小合格输出格式（新增硬规则）

正式回答必须显式带出以下 8 个二级标题，缺任一项都视为不合格：

```text
## pages.json 改动点
## 平台差异收口位置
## 请求封装与登录态
## 页面状态语义
## wot 引入与必要配置
## 样式 tokens 与密度
## 最小代码骨架
## 真机验证点与风险
```

- 若某项因信息不足无法展开，必须写“当前缺失信息 + 默认假设 + 风险”，禁止跳过标题。
- 回答结束前必须自检一次，确认 8 个标题都出现。

## 自检与失败回退（新增硬规则）

在输出最终答案前，必须按下列顺序做自检：

1. 是否已追问“页面路径或业务域 / 端能力 / 分包”。
2. 是否已明确 `pages.json` 变更点或说明无需变更的原因。
3. 是否已给出 `utils/platform.ts` 与 `utils/adapter/*` 的收口方案。
4. 是否已写明 401、网络错误、业务错误、登录失效的恢复动作。
5. 是否已明确 wot 采用 `uni_modules` 还是 npm。
6. 是否已写出至少 3 条真机验证点或风险点。

若任一步不满足：

- 不允许继续补长篇解释。
- 直接降级为“最小落地骨架 + 缺失信息 + 风险点”。
- 明确告诉用户哪些信息缺失，避免伪装成完整方案。

## 平台差异收口（硬规则）

- 平台判断必须集中：`utils/platform.ts`。
- 平台实现必须集中：`utils/adapter/*`，按能力组织（如 `storage.ts`、`share.ts`），不按端组织。
- 业务页面禁止出现 `#ifdef APP-PLUS` / `#ifdef MP-WEIXIN` 等条件编译。
- 仅在 `platform.ts` 与 `adapter/*` 内允许条件编译；若同一能力在 `adapter` 里超过 3 个端实现，必须抽公共逻辑。

例外策略：

- 若确实存在页面级渲染差异，允许在页面层做 1 处薄包装组件切换，例如 `BizMapPicker` / `BizShareEntry`。
- 即使有例外，也不允许在业务页面散写多段 `#ifdef`；必须把差异继续收敛到组件或 adapter。
- 若用户项目已存在 `src/platform/*`、`src/bridge/*` 等目录，必须做目录映射，不强迫改名。

## 请求与登录态（硬规则）

必须显式说明：

- 401 / 登录失效后的处理：跳登录页 or 静默续签，二选一并说明推荐策略。
- 跨端 cookie 处理差异：App 用持久化 cookie，微信小程序由 wx.request 自动管理；必须说明本项目选择哪一套。
- 错误语义至少分：网络错误、业务错误、登录失效；每种必须给恢复动作（重试/去登录/降级）。

若用户未答：默认推荐“小程序侧静默续签 + 必要跳登录；App 侧用统一 token + 401 跳登录”，并在风险里说明。

最小可交付要求：

- 必须说明 request 层文件落点，如 `src/utils/request.ts`。
- 必须说明 token / session 的持久化落点，如 `src/utils/adapter/storage.ts`。
- 必须说明登录失效后当前页是保留回跳地址，还是直接清空状态；两者必须选一套主策略。
- 若使用 Mock，必须说明 Mock 仅覆盖开发阶段，不得把 Mock 接口命名成正式 API 契约。

## 状态语义（按页面类型，硬规则）

- 列表页：loading / empty / error / 下拉刷新 / 触底加载；empty 区分无数据与查询无结果；error 必须有重试。
- 详情页：loading / error；empty 区分不存在与无权限。
- 表单页：提交 loading、校验错误、接口错误、离开页面未保存提示（若涉及编辑）。
- 设置页：尽量内联反馈，避免大弹窗。

## 样式与单位（硬规则）

- 设计稿基准：750rpx。
- 字号建议：辅助 24rpx、正文 28rpx、标题 32rpx。
- 间距建议：4 的倍数（16rpx/24rpx/32rpx）优先。
- 颜色与间距必须走 tokens 或 wot 变量层，禁止散写。

推荐最小落点：

- `src/styles/tokens.scss`：颜色、字号、间距、圆角、层级。
- `src/styles/index.scss`：全局样式入口、wot 变量桥接、通用 mixin。
- `uni.scss`：仅放全局变量桥与框架要求项，不承担业务样式堆积。

禁止项：

- 禁止页面内直接散写大段硬编码 `rpx`、颜色值和阴影值。
- 禁止把某个页面的视觉特例直接塞进 `uni.scss`，导致全局污染。

## 命名与目录

标准目录结构：

```text
src/
  api/
  components/
    base/
    biz/
  composables/
  pages/
  static/
  styles/
    tokens.scss
    index.scss
  utils/
    request.ts
    platform.ts
    adapter/
  App.vue
  main.ts
  manifest.json
  pages.json
  uni.scss
```

命名硬规则：

- 页面路径：业务域/页面名，或 kebab-case
- 组件：PascalCase
- wot 组件：保持官方 `wd-` 前缀
- composable：useXxx
- 平台判断文件固定：`utils/platform.ts`

## 存量项目兼容映射（新增硬规则）

若用户项目不是标准目录，不允许强迫其“先重构目录再回答”，而是必须做映射：

| 常见存量目录 | 视为规范目录 | 输出要求 |
| --- | --- | --- |
| `src/subpackages/*` | 分包页面目录 | 保持原分包结构，补 `pages.json` 注册点 |
| `src/services/*` | `src/api/*` | API 文件沿用原目录，同时补请求语义 |
| `src/common/wx/*` | `src/utils/adapter/*` | 按能力归并，不要求一次性迁移 |
| `src/mixins/*` | `src/composables/*` | 说明增量替换策略，不承诺全量改造 |

- 若项目已存在 `pages.config.ts`、插件化 pages 生成脚本或多端构建脚本，必须优先复用，不允许把 `pages.json` 写成唯一入口。
- 若项目已使用自定义 UI 库或 `uv-ui`，而用户只是问 uni-app 结构，不要强行把全站切去 wot；仅在明确要求 wot 时给迁移建议。

## 冲突仲裁规则

- 当“快”与“规范”冲突：优先保证 `pages.json` 落点 + 平台差异收口 + 状态语义；允许牺牲长代码与长教程。
- 当“跨端混提”与“单端稳定输出”冲突：必须拆分需求，禁止混答。
- 当“uni_modules 简单”与“npm 锁版严格”冲突：默认 uni_modules；只有用户明确 npm 时才切。

## 反例（禁止输出的样子）

- 只讲组件名：`用 wd-button、wd-cell 写就完了`。
- 假装跨端一致：`一套代码三端通用`而不提登录态/cookie 差异。
- 把 `#ifdef` 写满页面：业务页面内散写 `APP-PLUS` / `MP-WEIXIN`。
- 错误一句“自行兼容”：对 App/小程序/H5 差异不展开。
- 一上来要求用户先统一重构目录，再继续回答当前问题。
- 无真机验证点，只给静态代码片段，回避跨端能力差异。

## 最小正例（新增）

用户：`我要新增订单详情页，uni-app Vue3，小程序和 App 都要，已有 request 层，使用 wot`

合格回答必须至少满足：

- 第一段直接给出 `pages.json` 新增路由、分包位置或说明不分包。
- 平台差异里明确 `utils/platform.ts` 与 `utils/adapter/*` 落点，并说明哪些能力只在 App 生效。
- 请求层里明确 401、网络错误、业务错误、登录失效的恢复动作。
- 真机验证点里明确小程序进入页、App 返回页、弱网、登录过期、分享链路等场景。

## 输出后检查清单（给模型执行，不是给用户讲理念）

输出结束前逐项检查，任一缺失都必须重写：

- 是否包含 8 个固定标题
- 是否出现 `pages.json` 与平台差异落点
- 是否出现 401 / 网络错误 / 业务错误 / 登录失效处理
- 是否明确 wot 引入方式
- 是否出现 tokens 落点与禁止散写
- 是否出现真机验证点与风险
